% ── 1. INTRODUCCIÓN ──────────────────────────────────────────────────────────
\section{Introducción}

Las pruebas de software constituyen una disciplina fundamental en la ingeniería de software para garantizar la calidad, seguridad y corrección de los sistemas antes de su despliegue. Las estrategias de caja negra (\textit{black-box}) y caja blanca (\textit{white-box}) representan enfoques complementarios cuya combinación maximiza la detección de defectos \cite{myers2011art}.

El presente estudio empírico comparativo aplica ambos enfoques sobre una \textbf{API REST de Autenticación de Usuarios} desarrollada con ASP.NET Core Identity en .NET~10, cuya emisión de tokens JWT utiliza el algoritmo \textbf{ES256} (ECDSA P-256 / SHA-256) para firma y verificación de identidad. El objetivo es diseñar, ejecutar y analizar casos de prueba sobre los cuatro endpoints de autenticación, derivar métricas comparativas y emitir recomendaciones basadas en evidencia empírica.

La selección de este sistema responde a su representatividad en contextos de producción reales: el uso de criptografía asimétrica (ECDSA) para tokens JWT presenta rutas de código específicas que solo la cobertura estructural puede revelar, mientras que el comportamiento externo de los endpoints debe ser validado desde la perspectiva del cliente mediante técnicas de caja negra.

\section{Metodología}

\subsection{Descripción del Sistema Bajo Prueba}

\noindent\textbf{Tipo:} API RESTful — ASP.NET Core 10 con Identity, EF Core 10 (SQLite), JWT Bearer.

\noindent\textbf{Arquitectura de seguridad:}
El sistema genera un par de claves ECDSA P-256 (curva \texttt{nistP256}) en el arranque mediante \texttt{EcdsaKeyProvider}. Los tokens JWT emitidos tienen la siguiente anatomía:

\begin{lstlisting}[language={},caption={Estructura del JWT emitido (ES256)}]
Header:  { "alg": "ES256", "kid": "ecdsa-1", "typ": "JWT" }
Payload: {
  "iss": "https://authapi.local",
  "aud": "https://authapi.local/resources",
  "sub": "<userId>",           // Identity primary key
  "jti": "<guid>",             // unique token id (revocacion)
  "iat": 1700000000,           // issued-at (Unix epoch)
  "exp": 1700003600,           // expiry = iat + 60 min
  "email": "user@domain.com",
  "unique_name": "username",
  "display_name": "...",
  "email_verified": "true",
  "role": ["User"]
}
Signature: ES256(base64url(header) + "." + base64url(payload), ECDSA-P256-privKey)
\end{lstlisting}

\subsection{Definición de los Enfoques de Prueba}

\subsubsection{Pruebas de Caja Negra}

Evalúan el comportamiento del sistema exclusivamente a través de sus contratos HTTP: método, ruta, body, status code y payload de respuesta. No se tiene conocimiento del código fuente durante el diseño de casos.

\textbf{Técnicas aplicadas:}
\begin{itemize}[itemsep=2pt]
    \item \textbf{Partición de equivalencia (PE):} Clases válidas e inválidas por campo (email, password, username, token).
    \item \textbf{Análisis de valores límite (VL):} Casos en los extremos de las particiones (longitud mínima/máxima de password, token justo expirado vs.\ activo).
\end{itemize}

\subsubsection{Pruebas de Caja Blanca}

Diseñadas mediante inspección directa del código fuente. El criterio adoptado es \textbf{cobertura de ramas} (\textit{branch coverage}), que exige que cada rama de cada decisión condicional sea ejecutada al menos una vez. Se complementa con cobertura de sentencias como métrica base \cite{zhu1997software}.

Las decisiones críticas identificadas en el código son:

\subsubsection{Justificación del Uso de Ambos Enfoques}

Las pruebas de caja negra verifican el contrato HTTP del sistema; las de caja blanca garantizan cobertura de las rutas criptográficas y de control interno. La combinación es especialmente crítica aquí porque la lógica de blacklist (DEF-02) reside en una rama interna que la caja negra solo detecta como síntoma, sin localizar la causa raíz \cite{pressman2014software}.

% DEF-01 detail block
{\sloppy
\paragraph*{DEF-01 — Análisis y corrección aplicada}

\textit{Causa raíz:} \texttt{RegisterRequest} era un \textit{positional record} con parámetro
\texttt{string} non-nullable. El model binder de .NET~10 asigna \texttt{null} al campo ausente
del JSON sin activar \texttt{[Required]}; \texttt{FindByNameAsync(null)} retorna \texttt{null}
sin lanzar excepción, e Identity registra el usuario con \texttt{UserName = null}.
Los errores de validación retornaban \texttt{ProblemDetails} en lugar del \texttt{ErrorResponse}
propio de la API.

\textit{Corrección aplicada:} \texttt{RegisterRequest} convertido a \texttt{class} con dos capas:
\begin{itemize}[itemsep=1pt, topsep=2pt]
    \item \texttt{required} (C\# 11) — STJ lanza \texttt{JsonException} si el campo está ausente
          del JSON (capa de deserialización).
    \item \texttt{[Required]}, \texttt{[EmailAddress]}, \texttt{[StringLength]} — captura valores
          \texttt{null} o inválidos en validación de modelo.
    \item \texttt{InvalidModelStateResponseFactory} — unifica ambas rutas en
          \texttt{ErrorResponse("VALIDATION\_FAILED",~...)} en lugar de \texttt{ProblemDetails}.
\end{itemize}
}% end \sloppy

\vspace{0.3cm}

% DEF-02 detail block
{\sloppy
\paragraph*{DEF-02 — Análisis y corrección recomendada}

\textit{Causa raíz:} \texttt{OnTokenValidated} dispara \emph{después} de que el middleware JWT
ya construyó el \texttt{ClaimsPrincipal}. \texttt{ctx.Fail()} sin \texttt{return} explícito
no garantiza el corte del pipeline. El defecto es de \emph{composición}: \texttt{TokenBlacklist}
funciona correctamente en aislamiento (CB-13/CB-14 pasan al 100\%), pero su integración con
\texttt{JwtBearerEvents} no produce el rechazo HTTP~401 esperado.

\textit{Corrección recomendada:} mover el check a un middleware inline entre
\texttt{UseAuthentication()} y \texttt{UseAuthorization()}, que inspeccione
\texttt{HttpContext.User} ya construido y retorne HTTP~401 si el \texttt{jti} está revocado.
Alternativa mínima: agregar \texttt{return} tras \texttt{ctx.Fail()}.

\textit{Implicación académica:} 100\% de cobertura de ramas en tests unitarios no detecta
defectos de composición entre componentes. Esta clase de defecto solo es observable mediante
pruebas de caja negra de extremo a extremo.
}% end \sloppy

% ── 4. DISCUSIÓN ─────────────────────────────────────────────────────────────
\section{Discusión}

\subsection{Comparación de Resultados}

Los resultados revelan una complementariedad clara. Las pruebas de caja negra detectaron dos defectos de alto impacto (DEF-01 y DEF-02) observando únicamente los status codes HTTP y los payloads de respuesta. Los 25 tests de caja blanca (ejecutados y verificados el 1 de marzo de 2026) lograron \textbf{100\% de cobertura de ramas} en los módulos de criptografía (\texttt{EcdsaKeyProvider}, \texttt{TokenService}) y de revocación (\texttt{TokenBlacklist}), con 97.4\% de cobertura de líneas en \texttt{EcdsaKeyProvider} (el 2.6\% restante corresponde a la rama de PEM corrupto, fuera del alcance definido).

La divergencia más relevante es que \textbf{CB-14} (rama D5 verdadera en \texttt{TokenBlacklist}) pasa correctamente en unidad, mientras que \textbf{BN-16} falla en integración. Esto expone un defecto de composición: el componente funciona, pero su integración con el pipeline de JWT Bearer es incorrecta. Este tipo de defecto \emph{solo es detectable} mediante pruebas de caja negra de extremo a extremo.

En cuanto al rendimiento de ejecución automatizada, los 25 tests de caja blanca completaron en \textbf{2.507 segundos} (promedio 100 ms/test), lo que los hace aptos para ejecución en cada commit en un pipeline CI/CD. El tiempo dominante (\texttt{TokenServiceTests}: 1.851 s) se debe a la inicialización del fixture con \texttt{UserManager} y EF Core InMemory, no a la lógica de negocio bajo prueba.

\subsection{Contextos de Aplicación}

Para sistemas de autenticación con criptografía asimétrica, se recomienda: (1) caja blanca para verificar el algoritmo de firma, la curva ECDSA y la ausencia de material privado en la clave pública; (2) caja negra para validar el comportamiento de revocación, lockout y manejo de tokens malformados desde la perspectiva del cliente. En pipelines CI/CD, los tests de caja blanca (xUnit) se ejecutan en cada commit; los de caja negra (Postman/Newman) se ejecutan en el entorno de staging antes de cada despliegue \cite{cohn2009succeeding}.

% ── 5. CONCLUSIONES ──────────────────────────────────────────────────────────
\section{Conclusiones}

\begin{enumerate}[itemsep=4pt]
    \item \textbf{Defectos complementarios:} La caja negra detectó dos defectos de integración que la caja blanca no detectó. La caja blanca confirmó la corrección interna de los módulos criptográficos. Ningún enfoque por sí solo habría dado cobertura completa.

    \item \textbf{Criptografía exige caja blanca:} La verificación de que el algoritmo es ES256 (y no RS256), que la curva es P-256 y que la clave pública no contiene el escalar privado (\texttt{d}) \emph{solo} es posible mediante inspección estructural. La caja negra ve un token válido, pero no puede inferir qué algoritmo lo firmó.

    \item \textbf{Defectos de composición exigen caja negra:} El defecto de blacklist (DEF-02) reside en el orden de ejecución dentro del pipeline de ASP.NET Core, no en ningún componente individual. Solo pruebas de extremo a extremo lo revelan.

    \item \textbf{Cobertura 100\% de ramas no implica ausencia de defectos de integración:} Los tres módulos bajo prueba directa alcanzaron 100\% de cobertura de ramas (medido con Coverlet). Sin embargo, el defecto DEF-02 reside en la composición con el pipeline de ASP.NET Core (\texttt{JwtBearerEvents}), que no es alcanzado por los tests unitarios. La métrica de cobertura debe interpretarse junto con los resultados de pruebas de integración de extremo a extremo.

    \item \textbf{Corrección aplicada para DEF-01:} \texttt{RegisterRequest} fue convertido de \textit{positional record} a \texttt{class} con dos capas de defensa: \texttt{required} (C\# 11) bloquea campos ausentes en deserialización STJ; \texttt{[Required]} + \texttt{[EmailAddress]} + \texttt{[StringLength]} bloquean valores inválidos en validación de modelo. \texttt{InvalidModelStateResponseFactory} unifica ambas rutas de error en el formato \texttt{ErrorResponse} propio de la API. BN-08 ahora retorna HTTP~400 con \texttt{VALIDATION\_FAILED}.

    \item \textbf{Corrección recomendada para DEF-02:} Agregar \texttt{return} tras \texttt{ctx.Fail()} en \texttt{OnTokenValidated}, o mover el check de blacklist a un middleware inline entre \texttt{UseAuthentication()} y \texttt{UseAuthorization()} que inspeccione \texttt{HttpContext.User.FindFirstValue(Jti)} y retorne HTTP~401 directamente si el \texttt{jti} está revocado. Esta segunda opción es más robusta porque opera sobre el \texttt{ClaimsPrincipal} ya construido, independientemente del comportamiento interno del handler JWT.
\end{enumerate}
